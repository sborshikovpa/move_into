**Полный конспект**

Видео посвящено потокам в Python и GIL: Global Interpreter Lock, глобальной блокировке интерпретатора в CPython.

Главная мысль: потоки в Python не всегда дают настоящую параллельность. Для CPU-bound задач, то есть тяжелых вычислений на процессоре, GIL мешает потокам выполняться одновременно. Для I/O-bound задач, например записи в файл, чтения, сетевых операций, потоки могут дать реальный прирост.

**Что такое GIL**

GIL встроен в интерпретатор CPython. Его задача — обеспечить безопасный доступ потоков к объектам в памяти.

Потоки одного процесса работают с общей областью памяти. Это удобно, но опасно: разные потоки могут одновременно обратиться к одному и тому же объекту и создать гонку за ресурс. GIL решает эту проблему грубо, но эффективно: в большинстве случаев одновременно исполняется только один поток Python-кода, даже если у компьютера много ядер.

Из-за этого многопоточность в Python часто выглядит параллельной, но на деле потоки выполняются по очереди. Интерпретатор переключает их достаточно быстро, поэтому создается иллюзия одновременной работы.

**Зачем тогда GIL нужен**

GIL делает большинство объектов Python условно потокобезопасными без сложных механизмов синхронизации. Разработчику не нужно постоянно думать о защите каждого объекта памяти при доступе из нескольких потоков.

Альтернативой были бы более сложные и дорогие механизмы разделения данных, похожие на то, как данные разделяются между процессами. Но тогда потоки стали бы тяжелее и менее удобными.

**Эксперимент 1: CPU-bound задача**

Автор берет вычисление чисел Фибоначчи как пример нагрузки на процессор.

Он создает общий список `result`, куда разные потоки складывают вычисленные значения. Это демонстрирует работу потоков с одним объектом в памяти. Благодаря GIL список можно использовать безопаснее, чем если бы потоки одновременно неконтролируемо меняли память.

Дальше автор запускает вычисления с разным количеством потоков:

- 1 поток;
- 2 потока;
- 4 потока;
- 8 потоков;
- много потоков, вплоть до десятков.

Результат: время почти не улучшается. Иногда становится чуть быстрее или чуть хуже, но это скорее погрешность. Даже при увеличении количества потоков программа ведет себя так, будто работает один поток.

Вывод из эксперимента: для тяжелых вычислений потоки в CPython почти не ускоряют выполнение, потому что все упирается в GIL. Потоки фактически выстраиваются в очередь и исполняются по одному.

**Эксперимент 2: I/O-bound задача**

Второй пример связан с записью в файл.

Автор специально пишет функцию не самым оптимальным способом: файл открывается внутри цикла. Это сделано намеренно, чтобы лучше показать эффект снятия GIL при файловых операциях.

Идея такая же: есть общее количество строк, которое нужно записать, и эта работа делится на части между потоками. Затем измеряется время выполнения при разном количестве потоков.

Результат: в отличие от вычислений Фибоначчи, при работе с файлом увеличение количества потоков дает заметный прирост производительности. Рост нелинейный, но он есть. Можно подобрать некоторое оптимальное количество потоков, при котором время выполнения будет минимальным.

Почему так происходит: при I/O-операциях GIL может сниматься, а ответственность за безопасную работу с файлом берет на себя операционная система. Поэтому здесь появляется настоящая параллельность ожидания и выполнения операций ввода-вывода.

**Главные выводы**

Для CPU-bound задач потоки в Python обычно не дают настоящего распараллеливания. Если задача активно грузит процессор, например считает сложные математические вещи, потоки будут ограничены GIL.

Для CPU-bound задач лучше использовать процессы, то есть `multiprocessing`, потому что процессы имеют отдельную память и отдельные интерпретаторы. Это позволяет задействовать несколько ядер.

Для I/O-bound задач потоки подходят хорошо. Файлы, сеть, консольный вывод, ожидание внешних ресурсов — все это может ускоряться через многопоточность, потому что во время ожидания GIL часто освобождается.

Потоки дешевле процессов: ими проще управлять, они легче с точки зрения ресурсов и удобнее для многих практических задач.

Но не каждую задачу нужно масштабировать потоками или процессами. Для простых задач обычное последовательное выполнение может оказаться быстрее из-за отсутствия накладных расходов.

Некоторые библиотеки, особенно написанные на C/C++ или использующие нативные вычисления, могут временно освобождать GIL. Поэтому они способны достигать настоящей параллельности даже в задачах, похожих на CPU-bound. В субтитрах название библиотеки распознано не полностью, но смысл именно в том, что часть библиотек умеет обходить ограничение GIL на время своих внутренних вычислений.

Также GIL — это особенность CPython, стандартного интерпретатора Python. Теоретически от него можно уйти, используя другой интерпретатор Python, если он подходит под задачу.

**Итоговая формула**

- CPU-bound → лучше процессы, не потоки.
- I/O-bound → потоки могут быть полезны.
- Простая задача → иногда лучше вообще без параллелизма.
- GIL не “плохой” и не “хороший” сам по себе: он дает безопасность и простоту, но ограничивает настоящую многопоточность в CPython.

Коротко, суть примеров такая:

```
from threading import Thread
from time import perf_counter


def fib(n):
    if n < 2:
        return n
    return fib(n - 1) + fib(n - 2)


def worker(numbers, result):
    for n in numbers:
        result.append(fib(n))


numbers = [30] * 30
result = []

start = perf_counter()

threads = [
    Thread(target=worker, args=(numbers[i::4], result))
    for i in range(4)
]

for thread in threads:
    thread.start()

for thread in threads:
    thread.join()

print(perf_counter() - start)
```

Этот пример почти не ускорится от увеличения потоков, потому что вычисление Фибоначчи грузит CPU, а GIL дает исполняться только одному потоку Python-кода за раз.

Второй пример:

```
from threading import Thread
from time import perf_counter


def write_lines(start, end, filename):
    for i in range(start, end):
        with open(filename, "a", encoding="utf-8") as file:
            file.write(f"{i}\n")


total = 100_000
threads_count = 8
chunk = total // threads_count

start_time = perf_counter()

threads = [
    Thread(
        target=write_lines,
        args=(i * chunk, (i + 1) * chunk, "output.txt")
    )
    for i in range(threads_count)
]

for thread in threads:
    thread.start()

for thread in threads:
    thread.join()

print(perf_counter() - start_time)
```

Этот пример может ускоряться при увеличении числа потоков, потому что запись в файл — это I/O, и во время таких операций GIL часто отпускается.

То есть в ролике показываются не абстрактные рассуждения, а сравнение:

`Фибоначчи в потоках` → почти нет ускорения.  
`Запись в файл в потоках` → ускорение есть.
